# Package Manager -- Deep Dive

Vex's package manager (`vex pm`) handles dependency management, publishing, and workspace orchestration.

## `vex.toml` Manifest -- Full Schema

```toml
[package]
name = "my-project"
version = "0.1.0"
description = "A Vex project"
authors = ["Alice <alice@example.com>"]
license = "MIT"
repository = "https://github.com/alice/my-project"
keywords = ["web", "server"]
categories = ["network", "web"]

# Vex language version compatibility
[package.vex]
version = "0.1.2"

[dependencies]
http = "0.3.0"                    # exact version
serde = ">=1.0, <2.0"            # version range
crypto = { version = "0.5", features = ["aead", "hashing"] }
utils = { path = "../utils" }     # path dependency
my-lib = { git = "https://github.com/user/repo", branch = "main" }

[dev-dependencies]
testing = "0.2"

[build-dependencies]
codegen = "0.1"

[workspace]
members = ["core", "cli", "web"]

[workspace.dependencies]
# Shared dependency versions across workspace
serde = "1.0"
http = "0.3"

[profile.release]
opt-level = 3
lto = "thin"
debug-info = "none"
codegen-units = 1

[profile.bench]
opt-level = 3
lto = "full"

[target.x86_64-linux.build]
link-libs = ["pthread", "m", "dl"]

[target.aarch64-apple-darwin.build]
frameworks = ["Metal", "CoreFoundation"]

[target.wasm32]
# WASM-specific configuration
```

### Version Constraints

| Syntax          | Meaning                            |
| --------------- | ---------------------------------- |
| `"0.3.0"`       | Exactly 0.3.0                      |
| `">=1.0, <2.0"` | 1.0.0 to 1.x.x                     |
| `"^1.2.3"`      | >=1.2.3, <2.0.0 (compatible)       |
| `"~1.2.3"`      | >=1.2.3, <1.3.0 (minor-compatible) |
| `"*"`           | Any version                        |
| `"=1.0"`        | Exactly 1.0.0                      |

## Dependency Resolution

Vex uses a minimum-version resolution strategy (similar to Go modules):

1. Build the dependency graph from `vex.toml` files
2. For each package, select the **minimum** version that satisfies ALL constraints
3. If multiple packages require the same dependency at incompatible versions, **compile error** (no diamond dependency silently duplicated)
4. Resolution is deterministic and reproducible

### The `vex.lock` File

After resolution, the exact versions are recorded in `vex.lock`:

```toml
# Auto-generated -- do not edit manually
# Generated by vex pm resolve on 2026-06-05

[[packages]]
name = "http"
version = "0.3.4"
source = "registry+https://registry.vex-lang.org"
checksum = "sha256:abc123def456..."

[[packages]]
name = "serde"
version = "1.2.0"
source = "registry+https://registry.vex-lang.org"
checksum = "sha256:789abc123def..."

[[packages]]
name = "utils"
version = "0.1.0"
source = "path+../utils"
```

Commit `vex.lock` to version control for applications; omit for libraries (let consumers resolve).

## Workspaces

Workspaces orchestrate multiple packages in a single repository:

```
my-project/
├── vex.toml              # workspace root
├── vex.lock
├── core/
│   ├── vex.toml          # package: core
│   └── src/
├── cli/
│   ├── vex.toml          # package: cli (depends on core)
│   └── src/
└── web/
    ├── vex.toml          # package: web (depends on core + http)
    └── src/
```

Root `vex.toml`:

```toml
[workspace]
members = ["core", "cli", "web"]

[workspace.dependencies]
serde = "1.0"
```

Member `cli/vex.toml`:

```toml
[package]
name = "my-cli"
version = "0.1.0"

[dependencies]
core = { path = "../core" }
serde = { workspace = true }  # inherit from workspace
```

### Workspace Commands

```bash
vex pm build          # build all workspace members
vex pm test           # test all workspace members
vex pm fmt --check    # check formatting across workspace
vex pm update         # update all dependencies
```

## Publishing Packages

### Pre-Publish Checklist

```bash
# Ensure all tests pass
vex test

# Check formatting
vex fmt --check

# Verify package can be packaged
vex pm package --dry-run

# Check for common issues
vex pm lint
```

### Publishing

```bash
# Login to registry (one-time)
vex pm login --token YOUR_API_TOKEN

# Publish
vex pm publish

# Publishing automatically:
# 1. Packages source files
# 2. Computes checksum
# 3. Uploads to registry
# 4. Updates registry index
```

### Version Bumping

```bash
vex pm version patch     # 0.1.0 -> 0.1.1
vex pm version minor     # 0.1.0 -> 0.2.0
vex pm version major     # 0.1.0 -> 1.0.0
vex pm version 1.5.0     # Set specific version
```

## Registry Architecture

### Registry API

```
Base URL: https://registry.vex-lang.org

GET  /api/v1/packages/:name           # package metadata
GET  /api/v1/packages/:name/:version  # specific version
GET  /api/v1/packages/:name/download  # download .vx.tar.gz
POST /api/v1/packages/publish         # publish new version
GET  /api/v1/search?q=query           # search packages
```

### Package Index

The registry maintains a Git-based index for fast dependency resolution:

```
https://github.com/vex-lang/package-index
├── http
├── serde
├── crypto
└── ...
```

Each package file contains version metadata and checksums. `vex pm` clones/updates the index locally.

## Git Dependencies

```toml
[dependencies]
# From a Git repository
my-lib = { git = "https://github.com/user/repo" }

# Specific branch
my-lib = { git = "...", branch = "develop" }

# Specific tag
my-lib = { git = "...", tag = "v1.2.0" }

# Specific commit
my-lib = { git = "...", rev = "abc123def" }

# Subdirectory within repo (monorepo)
my-lib = { git = "...", subdir = "libs/my-lib" }
```

## Features and Conditional Dependencies

```toml
[package]
name = "my-lib"

[features]
default = ["std"]
std = []
gpu = ["sir"]
sir = []
tls = ["crypto/tls"]

[dependencies]
crypto = { version = "0.5", optional = true }
sir = { version = "0.1", optional = true }

# Consumer selects features:
# my-lib = { version = "1.0", features = ["gpu", "tls"] }
```

## Build Scripts

For complex build requirements, `build.vx` runs before compilation:

```vex
// build.vx -- executed at build time
fn main() {
    // Generate code
    let schema = readFile("schema.json")
    let code = generateFromSchema(schema)
    writeFile("src/generated.vx", code)

    // Link system libraries
    pkgConfig("openssl")
    pkgConfig("zlib")
}
```

## Caching

Package downloads are cached in `~/.vex/cache/`:

```
~/.vex/cache/
├── registry/
│   ├── http-0.3.4.vx.tar.gz
│   └── serde-1.2.0.vx.tar.gz
├── git/
│   ├── github.com-user-repo/
│   └── ...
└── index/
    └── package-index.git/
```

## Best Practices

1. **Commit `vex.lock` for applications, omit for libraries.**
2. **Use precise version constraints** in libraries (`">=1.0, <2.0"` rather than `"*"`).
3. **Run `vex pm update` regularly** to get security patches.
4. **Use workspaces** for multi-package repositories -- shared lockfile, unified commands.
5. **Pin Git dependencies to tags or commits** for reproducibility.
6. **Use features** to make heavy dependencies optional (GPU, crypto, TLS).
7. **Test with `--no-default-features`** to ensure optional dependencies truly are optional.
